Ecosystem

Margarita

Margarita templates bring structure and reusability to your prompts without the complexity of a full framework.

Explore Margarita Docs

Markdown Output

Anything between << and >> is plain Markdown

---
description: A template that demonstrates basic markdown formatting.
---
<<
# This is a normal heading
  - A list
A **bold** statement followwed by *italics* and a [link](https://www.banyango.com/margarita/latest).

Anything you can do in markdown you can do here.
>>
for item in items:
  << - This is also markdown -> ${ item } >>

Variable Substitution

Pass data in through json context on the CLI or directly via includes and use ${ variable } anywhere in the body.

# greet.mg
---
name: greet
description: A simple template that demonstrates variable substitution.
parameter1: name (string): The name of the person to greet.
parameter2: project (string): The name of the project or product.
---
<< Hello, ${ name }! >>
<< You are using ${ project }. >>

Conditionals

Use if var == value: / else: to include or exclude sections based on context. 

# access.mg
---
parameters: level (string): Access level of the user, e.g. "admin" or "user".
---
<< Access level: ${ level } >>
if level == "admin":
  << You have full access. >>
else:
  << Read-only mode. >>

Loops

Iterate over a list in the context with for item in list:. Each iteration renders the indented block — ideal for enumerating examples or steps.

# tasks.mg
---
description: A template that demonstrates looping over a list of tasks.
---
<< ## My Tasks >>
for task in tasks:
  << - ${ task } >>

Includes

Define reusable fragments in their own files and pull them into any template.

# components/greeting.mg
---
description: A simple greeting component that can be included in other templates.
parameters: user (string): The name of the person to greet. (metadata is not required but good practice!)
---
<< Hello, ${ user }! >>

# report.mg
---
description: A report template that includes a greeting component.
---
[[ components/greeting.mg user="Batman" ]]

CLI Tools

Render templates or extract metadata straight from the terminal. Supply context via a JSON file or let the template use its built-in defaults.

# render to stdout
margarita render greet.mg

# supply an external context file
margarita render greet.mg \
  --context ctx.json

# extract template metadata
margarita metadata report.mg

Python API

Import Margarita directly into Python for agentic loop patterns — render templates, inject dynamic context, and pipe the output straight into your LLM calls.

from margarita.parser import Parser
from margarita.renderer import Renderer

metadata, nodes = Parser().parse(template)
renderer = Renderer(context={
    "task": "Summarize the key points",
    "context": "User is researching AI agents"
})

prompt = renderer.render(nodes)

Meet Lime

Lime is an agent programming framework built directly on top of Margarita. It extends .mgx files with full agentic capabilities — giving you surgical control over the context, tools, and state sent to your LLM.

Explore Lime Docs

Run Agent

Use @effect run to execute the agent at any point in your .mgx file. Margarita renders all accumulated << >> blocks into the prompt before the agent is called.

# assistant.mgx
---
description: A simple Lime agent that answers a question.
---
// All statements are sequential

// 1. Add this to the context
<< You are a helpful assistant. What is the answer to life the universe and everything? >>

// 2. Run the agent
@effect run

// 3. Add more to the context
<< You need to think harder this time? >>

// 4. Run the agent again!
@effect run

Local Functions

Use @effect func to run a local Python function and store the return value in state — no LLM round-trip, no tokens consumed.

# processor.mgx
---
description: Call a Python function and store the result into state vars.
---
import summarize from module

// Run the summarize function and store into summary var
@effect func summarize("A really long text input") => summary

// Summary was stored into state vars so we can use it here!
<< Write a poem about the summary: ${ summary } >>

@effect run

Dynamic Tools

Use @effect tools to call a tool method directly from your .mgx file. The result is stored in state and available to the rest of the template. Use @effect tools clear to reset tool state.

# analyst.mgx
---
description: Fetch data with a tool, then pass it to the agent.
---
import fetch from module

// This registers your fetch Python function with the agent's tools.
@effect tools fetch(url: str) => data

<< You are a helpful assistant. You can fetch data from urls if you need... >>

@effect run
@effect tools clear

// Tools are cleared so no context is wasted on that tool.
<< Write a summary... >>
@effect run

Context Control

Use @effect context clear to reset the accumulated context between agent calls. Keeps each step focused and prevents context from ballooning across a multi-step workflow.

# multi-step.mgx
---
description: Two focused agent calls with a clean context boundary.
---
<< You are a code reviewer. >>
<< Review: ${ code } >>

@effect run
@effect context clear

<< You are a documentation writer. >>
<< Write docs for: ${ code } >>

@effect run

State

Values stored by @effect func and @effect tools persist for the lifetime of the run — templates and the agent can read from state at any point.

# stateful.mgx
---
description: Show how the agent state works.
---
@state count = 5
@state user = "Batman"

<<
The agent can get/set state variables using markdown.

Update the count variable to 10 and print out the user variable.
>>

@effect run

Meet Salt

Salt is a package manager for prompts. Install reusable prompt packages from a shared registry, lock every file to a verified hash, and protect your pipelines against prompt injection before they can do any damage.

📦

Package Registry

Install prompt packages from the Salt registry with a single command — just like npm for your LLM pipelines. Share and version your own packages for the whole team.

🔒

Lock Files

Pin the exact content hash of every installed prompt. A salt.lock file guarantees reproducible builds — the same template, every time, on every machine.

🛡

Injection Protection

Hash verification catches tampered templates before they reach your LLM. Any deviation from the locked hash fails fast — keeping malicious prompt injections out of your pipeline.

⚙

Built on Margarita

Salt packages are plain .mg and .mgx files — fully compatible with Margarita and Lime from day one. No new format to learn.

🔗

Link Hashing

Any link inside a << >> block is safely fetched at package time and hashed. If the page changes during execution, Salt notifies you — so tampered external context never silently poisons your context.
